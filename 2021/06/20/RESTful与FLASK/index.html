# RESTful

# RESTful 起源及设计理念

  1. 资源与 URI 和 URL
    1. 资源 resources 网络上的具体信息
    2. URI uniform resource identifier 统一资源标识符。用来唯一的标识一个资源
    3. URL uniform resouce locator 统一资源定位器，用来定位某个特定资源
  2. 表现层
    有多种表现形式
    1. 纯文本
    2. HTML
    3. JSON
  3. 状态转移 state transfer
    1. HTTP 协议，是一个无状态协议，意味着所有的状态需要保存到服务器
    2. GET (获取)、POST (增加)、PUT (增加)、DELETE (删除)
  4. REST 构架设计 6 原则
    1. Uniform Interface 统一接口
      包括设计客户端和服务端的接口，能使其独立更新修改升级
    2. Stateless 无状态
      应该向每次请求响应都是自包含，避免本次依赖上次，客户端对服务端的操作完全由表现层进行，即使没有把整个数据库表现出，也要允许通过 delete 删除资源，并且在一次操作中完成
    3. 可缓存的
      可以提高客户端的访问体验，还可以减少服务端的流量压力
    4. 服务端 客户端分离
      客户端不包括数据，服务端不包括用户状态，增加服务稳定性，可以对任意一端进行升级
    5. 客户端不直接连接服务端，可以通过其他链接，客户端也可以是某一个服务端，作为中间的服务端请求另一个服务端
    6. 按需编码，由于两端只通过接口通信，可以使用不同语言

# Flask

微型框架，表示保持核心简单，但是可扩展，Flask 不会替你做决定，数据库，引擎都任意，Flask 可以变成你任何想要的东西，不会变成任何你不想要的东西，一切由你做主
